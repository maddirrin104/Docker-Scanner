D12 - Docker Scanner
Công cụ quét Docker image

Python, Trivy, Docker API

Nhóm thực hiện: Võ Minh An, Nguyễn Gia Bảo, Bùi Nguyễn Công Hiếu, Sit Khải Đông
1

Abstract

  • Việc kiểm tra bảo mật các Docker image là bước quan trọng để đảm bảo an toàn khi triển khai ứng dụng trên môi trường container. Tuy nhiên, thao tác quét lỗ hổng với các công cụ có sẵn thường phức tạp và khó tích hợp vào quy trình DevOps.
  • Đề tài xây dựng một công cụ CLI bằng Python, đóng vai trò là một wrapper cho Trivy, giúp người dùng dễ dàng quét và tổng hợp các lỗ hổng bảo mật (CVE) trên Docker image dưới nhiều định dạng. Kết quả được hiển thị rõ ràng dưới dạng bảng, hỗ trợ tổng hợp, lưu trữ và đánh giá trực quan mức độ rủi ro.
  • Đơn giản hóa quy trình kiểm tra bảo mật, tăng tính minh bạch và hỗ trợ phát hiện, xử lý lỗ hổng sớm trong quá trình phát triển/phân phối phần mềm.
2

Introduction

  • Kiến trúc: Dùng Trivy làm công cụ xử lý lõi, xây dựng một wrapper bằng Python với các option phù hợp để sử dụng các thư viện có sẵn trong Trivy như Docker API, SQLite, JSON/YAML parsing, table formatting, logging,... nhằm đơn giản hóa thao tác quét lỗ hổng và hiển thị báo cáo.
  • Chức năng cốt lõi: Phân tích một Docker image (từ local hoặc remote registry) và liệt kê các gói phần mềm được cài đặt bên trong. Sử dụng một cơ sở dữ liệu lỗ hổng công khai (ví dụ: NVD) để đối chiếu các phiên bản gói phần mềm và tìm ra các CVE tương ứng. Sau đó xuất báo cáo dưới định dạng tùy chọn (JSON, HTML, CLI,...)
  • Dữ liệu & UI: Trivy tool, Vulnerabilities database (thường tích hợp sẵn trong tool), Python wrapper hỗ trợ thao tác và xuất báo cáo.
3

Methodology

  • Sử dụng Trivy làm công cụ quét lỗ hổng bảo mật cho Docker image, đảm bảo kết quả chính xác và cập nhật từ cơ sở dữ liệu CVE công khai.
  • Xây dựng một Python CLI wrapper để tự động hóa việc gọi Trivy, xử lý đầu ra JSON và tổng hợp thông tin lỗ hổng dưới dạng bảng dễ đọc.
  • Quy trình: Người dùng nhập tên Docker image → Wrapper gọi Trivy để quét → Kết quả được phân tích và hiển thị trực quan trên terminal (hoặc định dạng khác có thể tùy chỉnh trong command ở CLI).
  • Đánh giá hiệu quả dựa trên khả năng phát hiện lỗ hổng, mức độ tự động hóa và tính dễ sử dụng của công cụ.
4

System Architecture

Luồng dữ liệu: ① Contracts emit events → ② UI dùng Direct Log Reader (WS + getLogs) để đồng bộ → ③ UI duy trì Client Cache (memory/IndexedDB) dựng lại sổ lệnh → ④ UI gọi eth_call các hàm view (getBestBid/Ask, getBook(offset,limit), getOrder) để xác thực/hoàn thiện dữ liệu trước khi render.

Conclusion

  • Đã làm được: Giao diện đã hiển thị sổ lệnh (bids/asks, độ sâu theo cấp) và bảng Market theo cặp; dữ liệu đang đọc trực tiếp on-chain/seed. Chưa mở chức năng giao dịch (nạp/rút, đặt/hủy, khớp).
  • Giới hạn hiện tại: Chưa có Vault (nạp/rút), chưa có đặt/hủy/khớp lệnh, chưa dùng Subgraph nên realtime và truy vấn độ sâu còn hạn chế.
  • Lộ trình: (1) Xây Subgraph (Pair/Order/Level/Balance; đồng bộ events) → (2) Hoàn thiện Vault & luồng nạp/rút → (3) Bật đặt/hủy & khớp partial/full trên OrderBook → (4) Bảo mật CEI/ReentrancyGuard/AccessControl & bộ test → (5) Tối ưu gas/UX và triển khai testnet.
